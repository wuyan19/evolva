## Context

Evolva 是一个自修改应用，用户通过自然语言指令让 LLM 生成 JavaScript 代码来修改 UI。当前导入导出功能基于 DOM 快照（`outerHTML`），这种方式丢失了事件监听器和 JavaScript 行为。

核心观察：每次 LLM 回复的代码是基于完整当前 DOM 生成的完整版本代码，而非增量补丁。因此代码片段序列天然构成版本历史，最新版本即为当前应用的完整代码。

当前前端通过 Tauri 插件（`tauri-plugin-dialog`、`tauri-plugin-fs`）直接操作文件系统，导入导出逻辑散落在前端 JS 中。

## Goals / Non-Goals

**Goals:**
- 将导入导出逻辑从前端迁移到 Rust 后端
- 以 LLM 生成的代码版本为导出单位，而非 DOM 快照
- 支持版本选择回放，用户可编辑 JSON 文件切换 `active_version`
- 导入前检测未保存版本并提示确认

**Non-Goals:**
- 不实现版本 diff 或可视化比较
- 不实现云端同步或分享功能
- 不修改 LLM 调用流程或 system prompt
- 不实现多个 APP 的会话区分（一会话一导出）

## Decisions

### 1. 版本数据存储在 Rust 内存中

**决定**：使用 `tauri::State` 管理 `Mutex<Vec<Mutation>>` 在内存中累积版本。

**理由**：
- 单会话场景，无需持久化到磁盘（应用关闭即丢失，用户需主动导出）
- 内存操作简单、快速，无需引入数据库依赖
- `Mutex` 保证线程安全

**备选**：每次写入文件 → 过度 I/O，且需要管理临时文件清理。

### 2. 导出文件格式为 JSON

**决定**：使用 JSON 格式，文件扩展名 `.evolva.json`。

**理由**：
- JSON 可读性好，用户可直接编辑 `active_version` 字段选择回放版本
- Rust 生态中 `serde_json` 成熟稳定
- `.evolva.json` 扩展名既表明归属又保持 JSON 可读性

### 3. 导入时清除非核心 DOM 元素

**决定**：导入前移除所有 `id !== 'evolva-core'` 的 body 子元素和动态添加的 head style 标签。

**理由**：
- LLM 生成的代码基于"干净"的 Evolva 环境执行
- 残留 DOM 元素会导致代码操作意外元素或样式冲突

### 4. 确认对话框使用前端自定义模态框

**决定**：当存在未保存版本时，使用前端自定义模态框提示用户确认。

**理由**：
- 保持 UI 风格一致
- Tauri 原生对话框样式与应用主题不匹配
- 自定义模态框可以在暗/亮主题下正确显示

### 5. 后端提供 mutation 计数查询

**决定**：新增 `get_mutation_count` 命令，返回当前累积的版本数量。

**理由**：前端需要判断是否存在未保存版本来决定是否弹出确认框。

## Risks / Trade-offs

- **[版本重放失败]** → LLM 代码可能依赖特定的运行时状态（如 `topZ` 变量值）。缓解：确保 `setupDraggable`、`bringToFront`、`topZ` 等始终作为全局变量可用。
- **[导出文件被手动编辑破坏]** → JSON 格式便于用户编辑，但也可能导致格式错误。缓解：导入时做好 JSON 解析错误处理，给出明确的错误提示。
- **[内存中版本丢失]** → 应用崩溃或关闭时未保存的版本将丢失。缓解：这是可接受的行为，与"用户需主动导出"的设计一致。
